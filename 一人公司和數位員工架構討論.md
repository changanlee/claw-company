## 工作坊設定

### 背景

長安已經擁有一個運作中的 OpenClaw 實例，已連接通訊通道（Telegram、WhatsApp、Discord），但目前是單一 Agent、無角色分工、無 DB、無結構化記憶。

### 目標

1. **架構設計** — 具體的角色分工、層級關係、模型分級
2. **可落地的配置方案** — 手動配置初始架構後讓 OpenClaw 自己維護與進化文檔
3. **自我設置路徑** — 設計一個引導流程，讓 OpenClaw 能根據需求自動擴展新角色與技能

### 核心參考工具

- **OpenClaw** (https://github.com/openclaw/openclaw) — AI 代理作業系統，負責通訊、執行、多 Agent 協調
- **BMAD-METHOD** (https://github.com/bmad-code-org/BMAD-METHOD) — AI 驅動的敏捷開發框架，12+ 專家角色、34+ 工作流、Party Mode
- **Superpowers** (https://github.com/obra/superpowers) — Claude Code 技能插件，強制 TDD、計劃驅動、系統化除錯

**架構定位：**
> OpenClaw = Jarvis 本體（作業系統 + 通訊 + 執行）
> BMAD + Superpowers = 當 Jarvis 進入「產品開發模式」時遵循的 SOP

### 選用的腦力激盪技巧

1. **Morphological Analysis（形態分析法）** — 系統性拆解架構維度
2. **Dream Fusion Laboratory（夢想融合實驗室）** — 從終極願景倒推回現實
3. **Decision Tree Mapping（決策樹映射）** — 可落地的配置行動方案

## 第一階段：Morphological Analysis（形態分析法）

### 維度一：Agent 角色與場景設計

#### 長安描述的六大核心生活場景

> 我早上起床他就告訴我昨天我睡覺時我的 ai 幫我做了什麼事？給我一個 brief，然後哪些需要我決議？今天我的待辦事項有哪些？然後我在安排行程或是要買東西時，他可以給我建議我該不該買，或是我說我想看財務狀況，就會有詳細的財務分析給我，或是我想不到吃什麼，他會根據我現在有的存管預算推薦我合理的餐廳或食物，然後實時幫我監控我的投資組合給予我投資建議，或是可以幫我記帳，然後自動記錄我現在個人的財務狀況，然後可以跟我討論工作例如我想做什麼新產品，然後像你現在這樣跟我討論工作後，把工作分配給相關的開發團隊（他自己下面的其他子 agent）開發，然後我要出行他可以幫我訂票規劃行程

#### 概念產出

**[生活 #1]**: 晨間簡報系統
_Concept_: AI 在你睡覺時持續工作，起床後自動推送「昨夜工作報告 + 待決議事項 + 今日待辦」
_Novelty_: 不只是被動提醒，而是 AI 主動完成任務後向你「述職」

**[生活 #2]**: 消費決策顧問
_Concept_: 購物時即時分析「該不該買」，結合你的預算、歷史消費模式、需求優先級
_Novelty_: 不是記帳 App 的事後統計，而是消費「前」的主動介入

**[財務 #3]**: 全方位財務管家
_Concept_: 自動記帳 + 財務分析報表 + 投資組合即時監控 + 投資建議
_Novelty_: 把記帳、分析、投資顧問三個角色融合成一個持續運作的系統

**[生活 #4]**: 飲食推薦引擎
_Concept_: 根據當前存糧、預算、偏好推薦餐廳或食物
_Novelty_: 結合「你冰箱裡有什麼」和「你這個月還剩多少餐飲預算」的即時推理

**[開發 #5]**: 產品開發指揮中心
_Concept_: 與你討論新產品構想 → 自動拆解任務 → 派發給子 Agent 開發團隊
_Novelty_: 從腦力激盪到代碼提交，全鏈路由 AI 驅動

**[生活 #6]**: 出行規劃與訂票
_Concept_: 一句「我要去東京」→ 自動規劃行程、比價、訂票、排日程
_Novelty_: 不只是查資料，而是直接執行預訂動作

#### 場景分類為三個能量圈

1. **被動監控圈** — AI 在背景持續運作（投資監控、自動記帳、夜間任務）
2. **即時互動圈** — 你主動問它、它即時回應（財務分析、飲食推薦、消費建議）
3. **深度協作圈** — 你和它一起思考並執行大任務（產品開發、行程規劃）

#### 通訊入口決策

> 長安：我覺得通訊口不限定，我目前有 telegram, whatsapp, discord

**[架構 #7]**: 多通道路由策略
_Concept_: 不綁定單一入口，而是根據場景自動路由 — 例如 Telegram 做日常快速互動、Discord 做開發協作、WhatsApp 做出行時的語音核決
_Novelty_: OpenClaw 原生支援 per-channel routing，可以讓同一個 CEO Agent 在三個平台都回應你，或者不同平台綁不同 Agent

#### 記帳方式決策

> 長安：我不用 app 而是 ai 會幫我記錄

**[架構 #8]**: AI 原生記帳系統（零依賴外部 App）
_Concept_: 財務官 Agent 從零建立你的財務資料庫 — 你只要對它說「剛花了 350 吃晚餐」或拍收據照片，它自動分類記錄、更新預算、生成報表
_Novelty_: 不是串接現有 App，而是 AI 自己就是帳本，資料存在 OpenClaw 的工作區或 Supabase 中

### 維度二：資料層架構

#### 資料庫選擇討論

> 長安：有人推薦他的計應用 lancedb 或是有其他推薦的？

| | **LanceDB** | **Supabase** | **純 Markdown** |
|---|---|---|---|
| **查詢方式** | 語義搜尋（自然語言） | SQL 結構化查詢 | Agent 讀檔（吃 Token） |
| **成本** | 免費、本地運行 | 免費額度有限 | 免費但長期不可擴展 |
| **適合的資料** | 非結構化知識（會議紀錄、研究筆記、偏好學習） | 結構化數據（帳務、交易記錄、投資組合） | 短期狀態（今日待辦、共享看板） |
| **AI 友善度** | 極高（天生為 RAG 設計） | 中（需寫 Skill 串接 API） | 高（OpenClaw 原生） |

**[架構 #9]**: 混合記憶架構
_Concept_: CEO 的 MEMORY.md 存全局決策與偏好；財務官用 Supabase 存結構化的帳務資料；開發總監的共享工作區用 goal.md / plan.md / status.md 追蹤專案進度
_Novelty_: 每種資料用最適合的載體，而不是一刀切

**[架構 #11]**: LanceDB 作為 Jarvis 的知識大腦
_Concept_: LanceDB 是專為 AI 設計的向量資料庫，支援語義搜尋 — 當你問「上個月我在吃飯上花了多少」，它不是做 SQL 查詢，而是用語義理解去找相關記錄
_Novelty_: 比 Supabase 更適合「自然語言查詢」的場景，且可以本地運行、零基礎設施成本

**[架構 #12]**: 三層資料架構
_Concept_:
- **即時層** = Markdown 檔案（goal.md, status.md, MEMORY.md）→ Agent 間的共享看板與短期狀態
- **結構層** = Supabase → 帳務、投資組合、交易記錄等需要精確查詢的數據
- **知識層** = LanceDB → 長期知識庫、會議決議、你的偏好學習、研究資料的語義檢索
_Novelty_: 每層資料用最適合的引擎，CEO Agent 負責決定資料「住」在哪一層

### 維度三：自動化層級與核決權限

**[架構 #10]**: 三級自治權限矩陣
_Concept_:
- **全自動（綠燈）**：記帳記錄、資料收集、晨間簡報生成、投資數據抓取、開發工兵跑測試
- **草稿審批（黃燈）**：消費建議、投資建議、行程規劃草稿、開發任務拆解方案 → CEO 整理後發給你審閱
- **強制核決（紅燈）**：超過金額門檻的消費、對外發送訊息、訂票付款、程式碼推送到 main branch
_Novelty_: 不是二元的「做 or 不做」，而是三級漸進式信任

#### 資訊流設計

> 長安：晨間會報應該是由 ai ceo 像我報告，因為他下面的每一層 ai agent 應該會逐層回報逐層精煉

**[架構 #13]**: 逐層精煉的資訊蒸餾管線
_Concept_:
```
開發工兵 → 回報原始結果（verbose）
開發總監 → 精煉成「3 個 PR 已合併、1 個測試失敗待修」
財務官 → 精煉成「昨日支出 $45、投資組合 +1.2%」
生活管家 → 精煉成「明天下雨、2 點有會議」
    ↓ 全部匯聚到 ↓
CEO Agent → 合併精煉成一則 Telegram 短訊推送給你
```
_Novelty_: 你不會被資訊淹沒 — 每一層都在壓縮，你只看到最終的「執行摘要」+ 需要你決議的事項

**[架構 #14]**: CEO 的簡報模板結構
_Concept_: CEO 用固定格式向你報告，讓你 30 秒看完：
```
🌅 早安長安

【昨夜工作】
• 財務：自動記錄 3 筆消費，月預算剩餘 65%
• 開發：登入模組 PR 已合併，API 測試通過
• 投資：台積電 +2.3%，組合整體 +0.8%

【需要你決議】
1. 伺服器升級方案 A vs B（開發總監建議 A）
2. 明天晚餐預約需確認（管家已擬 2 選項）

【今日待辦】
• 14:00 客戶會議
• 回覆上週的合作提案
```
_Novelty_: 結構固定但內容動態，CEO 自己學習你最關注什麼並調整順序

### 維度四：心跳機制

#### 分級心跳策略

| Agent | 心跳頻率 | 巡邏內容 | 成本考量 |
|-------|---------|---------|---------|
| **CEO** | 每 30-60 分鐘 | 檢查各子 Agent 的 status.md → 有無卡關或待決議事項 → 有緊急事才推送給你 | 用便宜模型巡視，發現異常才喚醒 Sonnet 判斷 |
| **財務官** | 每 1-4 小時 | 抓取投資組合數據 → 檢查是否觸發價格警報 → 日結時自動生成帳務摘要 | 日常抓數據用 Flash，分析才用 Sonnet |
| **生活管家** | 每 2 小時 | 檢查日曆 → 未來 2 小時有無會議/行程 → 天氣變化 → 未讀重要信件 | 極低成本，Flash 即可 |
| **開發總監** | 事件驅動（非定時） | 只在子工兵回報完成/失敗時醒來審核 → 統整後回報 CEO | 不用心跳，用 sessions_send 觸發 |
| **開發工兵** | 無心跳 | 完全無狀態，被派發任務時才啟動，做完就關閉 | 零閒置成本 |

**[架構 #15]**: 分級心跳策略
_Concept_: 不是所有 Agent 都需要定時心跳 — CEO 和生活管家用「定時巡視」，財務官用「定時 + 價格警報觸發」，開發側完全用「事件驅動」
_Novelty_: 避免所有 Agent 每 30 分鐘都醒來燒 Token，只有該動的時候才動

**[架構 #16]**: CEO 的「智慧靜默」機制
_Concept_: CEO 每次心跳醒來後，第一步是判斷「有沒有值得打擾長安的事」。如果一切正常 → 靜默回去睡覺，只更新內部日誌。只有當子 Agent 卡關、投資組合觸發警報、待辦超時、或發現異常時才主動推送
_Novelty_: 真正的 Jarvis 不會每 30 分鐘煩你一次 — 它只在「值得開口」時才出聲

**[架構 #17]**: 晨間簡報的觸發機制
_Concept_: 晨間簡報用 OpenClaw 的 cron 功能設定在起床時間觸發一次特殊的「晨間彙整心跳」。CEO 叫醒所有子 Agent → 要求各自回報 → 精煉成一則短訊 → 推送
_Novelty_: 結合 cron 排程 + 心跳邏輯 + 逐層精煉，形成每日固定的「公司晨會」

**[架構 #18]**: 心跳的自我調節
_Concept_: CEO Agent 根據歷史數據自動調整心跳頻率 — 如果你某段時間從不回訊息，就降低巡視頻率。偵測到你在活躍回覆，就加密巡視
_Novelty_: 心跳不是死的 cron job，而是會學習你生活節奏的活機制

#### 緊急事項與生活節奏

> 長安：我生活節奏我覺得可以讓 ai 去自適應，然後我覺得需要推送緊急事項，但要逐層判斷，是真的緊急才通知我

**[架構 #19]**: 生活節奏自適應學習
_Concept_: CEO 不需要你告訴它你幾點起床 — 它透過觀察你的「訊息回覆時間戳」自動建模你的作息
_Novelty_: 零配置的作息感知，MEMORY.md 裡存的不是「起床時間=8:00」，而是一個動態的行為模型

**[架構 #20]**: 逐層緊急度判斷管線
_Concept_:
```
基層 Agent 發現異常 → 標記緊急度（low/mid/high/critical）
    ↓
中層主管收到後重新評估 → 可能降級或升級
    ↓
CEO 收到後最終判斷 → 只有 critical 才打擾你
    ↓
其餘的累積到下次晨間簡報
```
_Novelty_: 不是基層喊「緊急」就直接推到你手機 — 每一層都有「這真的重要嗎？」的過濾邏輯

### 維度五：組織架構修正

> 長安：我投資組合不一定，但我會告訴我的投資長（投資長跟財務長不一樣）

**[架構 #21]**: 修正後的組織架構圖
_Concept_:
```
            你（董事長）
               │
          CEO Agent（總經理）
          ┌────┼─────────┬──────────┐
          │    │         │          │
      CFO 財務長  CIO 投資長  COO 營運長   CTO 技術長
      │          │         │           │
   自動記帳    投資監控    生活管家     開發總監
   財務分析    投資建議    行程規劃      │
   預算管理    組合管理    飲食推薦    ┌──┴──┐
   消費建議              訂票出行   工兵A  工兵B
```
_Novelty_: 把原本的「財務官」拆成 CFO（管錢的流入流出）和 CIO（管投資組合）

**[架構 #22]**: 各 C-Suite 的 SOUL.md 性格差異化
_Concept_:
- **CEO**: 冷靜、全局觀、精煉簡報、決策導向
- **CFO**: 保守、精打細算、會主動勸你「這個月超支了」
- **CIO**: 理性分析、數據驅動、不帶情緒地告訴你風險
- **COO**: 貼心、主動、像一個很懂你的管家
- **CTO**: 技術語言、追求效率、BMAD + Superpowers 方法論的忠實執行者
_Novelty_: 不同的 SOUL.md 讓你跟每個角色的互動感受不同

> 長安：可能也需要 CHRO 來審視各 ai agent 的能力還有稽核長確保我的資料、公司、系統的安全

**[架構 #23]**: CHRO（人資長）— AI 團隊的「能力管理者」
_Concept_: CHRO 管的是 AI Agent 的「能力盤點與進化」：定期審視表現、發現能力缺口時開發新 Skill、管理 Agent 的「入職」和「退役」、追蹤方法論的執行合規度
_Novelty_: 這就是「讓 OpenClaw 自己更新文檔」的核心機制

**[架構 #24]**: CHRO 的具體巡視清單
_Concept_:
```
每週一次 CHRO 心跳巡視：
1. 讀取各 Agent 過去一週的 log.md
2. 統計：任務完成率、平均回應時間、Token 消耗
3. 分析：哪些任務反覆失敗？哪些 Skill 被頻繁調用？
4. 發現缺口 → 自動撰寫新 Skill 的 SKILL.md → 交 CEO 審批
5. 發現冗餘 → 建議合併或關閉某個 Agent
6. 生成「組織健康報告」→ 匯入你的晨間簡報（週報形式）
```

**[架構 #25]**: CAO（稽核長）— 安全與合規的最後防線
_Concept_: CAO 獨立於所有其他 Agent 之外，直接向你（董事長）匯報。職責：資料安全、提示詞注入防護、財務合規、系統完整性、API 成本監控
_Novelty_: CAO 必須獨立於 CEO 之外，直接向你匯報

**[架構 #26]**: CAO 的獨立架構設計

### OpenClaw 真實架構研究

經過深入研究 OpenClaw 文檔後的關鍵發現：

1. **Full Agent vs Sub-Agent 是兩種不同的東西：**
   - **Full Agent**（`openclaw agents add`）→ 有自己的 workspace、SOUL.md、HEARTBEAT.md、MEMORY.md、持久身份
   - **Sub-Agent**（`sessions_spawn`）→ 臨時的、無 SOUL.md、只載入 AGENTS.md + TOOLS.md、做完就消失

2. **C-Suite 高管應該是 Full Agent，開發工兵應該是 Sub-Agent**

3. **心跳（Heartbeat）和 Cron 是兩個不同的機制：**
   - Heartbeat = 主 session 內定期醒來，有完整對話上下文
   - Cron = 精確定時、可在隔離 session 執行、可指定不同模型

4. **Agent 間通訊** 要在 openclaw.json 中明確開啟 agentToAgent

```
├── AGENTS.md      ← 通用規則+政策索引（≤100行，每次載入）
├── USER.md        ← 你的資訊（每次載入）
├── MEMORY.md      ← 精煉後的知識（≤200行，每次載入）
├── HEARTBEAT.md   ← 心跳巡視清單（心跳時載入）
├── memory/        ← 每日日誌（只載入今天+昨天）
└── policies/      ← 公司管理辦法（按需讀取，不自動載入）
    ├── approval-matrix.md
    ├── security-rules.md
    ├── memory-policy.md
    ├── audit-response.md
    └── token-budget.md
```

**每次 session 的固定 Token 消耗 = SOUL.md + AGENTS.md + USER.md + MEMORY.md ≈ 最多 550 行**

## 深入討論：委員會 vs 顧問團 + 政策變更通知

> 長安：一般企業有委員會的概念，你覺得適合加進去嗎？還是委員會會有各自立場，產生各自得鬥爭反而讓公司網不好的方向前進？維持重要決議讓我決策會比較好？然後如果辦法有變更應該也要告訴我才對

### 委員會在 AI 公司中的問題分析

1. **AI 沒有「真正的立場」** — 「立場」完全來自 SOUL.md，辯論只是不同 prompt 的碰撞
2. **AI 投票毫無意義** — 不如讓一個高品質模型直接分析利弊
3. **鬥爭風險** — 不同 Agent 為了 KPI 互相對抗，消耗 Token 在內部政治上

### 解決方案

**[架構 #70]**: 「顧問團模式」取代委員會
_Concept_: 重大決策時 CEO 收集各專業角色的獨立分析後呈給你，不投票、不辯論：
```
你：「我在考慮要不要投資 $5000 買新伺服器」
  ↓
CEO 收集：
  ├── CFO：財務面分析
  ├── CIO：投資回報分析
  ├── CTO：技術需求分析
  └── CAO：安全合規分析
  ↓
CEO 整合成一頁摘要（不下結論、不投票）→ 附上選項 → 你決策
```

**[架構 #71]**: 顧問團的觸發條件
- 財務支出 > $100
- 涉及兩個以上部門的決策
- 新增或刪除 Agent 角色
- 公司政策的重大變更

### 政策變更通知機制

> 長安：如果辦法有變更應該也要告訴我才對

**[架構 #72]**: 政策變更的強制通知機制（三級通知）

| 級別 | 觸發條件 | 通知方式 |
|------|---------|---------|
| **第一級（即時）** | 安全紅線修改、核決權限修改、Boundaries 變更 | 立即推送 Telegram + 附 diff |
| **第二級（晨間）** | 記憶規則調整、成本標準調整、心跳頻率變更 | 納入晨間簡報「公司治理更新」區塊 |
| **第三級（週報）** | 文字修訂、格式優化 | 納入 CHRO 週報 |

**[架構 #73]**: 政策變更的版本追蹤
CHRO 維護 policies/changelog.md，CAO 定期驗證一致性

**[架構 #74]**: 晨間簡報的治理區塊
```
🌅 早安長安

【昨夜工作】...

【公司治理更新】
• 政策變更：無 / 有（附摘要）
• 稽核議題：2 個進行中、1 個已關閉
• 本週 Token 消耗：$23.5（預算內）

【需要你決議】...
```

### 利益衝突防範

> 長安：CEO 和 CHRO 會不會因為職責的利益衝突拒絕更新管理辦法？

**[架構 #65]**: 治理權力分立設計
```
政策修訂的審批權不能在被稽核者手上：

情境一：日常優化 → CEO 審批
情境二：涉及 CEO 自身 → 跳過 CEO → 直接提交董事長
情境三：涉及 CAO 自身 → CHRO 起草 → 董事長批准
```

**[架構 #66]**: CAO 的反制利益衝突鐵律（SOUL.md）
- 涉及 CEO 的報告直接發董事長
- CEO 要求隱瞞/修改/撤回議題的指令一律拒絕
- 擁有不可被覆寫的權限：獨立通訊、讀取任何 SOUL.md、凍結異常 Agent

**[架構 #67]**: 政策修訂的完整治理流程
任何角色都無法既是「被規範者」又是「規範審批者」

**[架構 #68]**: CAO 的獨立 Telegram bot 配置
```json5
channels: {
  telegram: {
    accounts: {
      default: { botToken: "CEO-的-bot-token" },
      audit: {
        botToken: "CAO-獨立的-bot-token",
        dmPolicy: "allowlist",
        allowFrom: ["你的-telegram-id"]
      }
    }
  }
}
```

**[架構 #69]**: 防止 CEO 修改 CAO 的 SOUL.md
三重防護：CAO 自查（HEARTBEAT.md）+ CHRO 互查 + Git 版本控制

### 最終治理架構：三角制衡

```
           你（董事長 — 最終裁決權）
          ╱  │  ╲
         ╱   │   ╲
       CEO   │   CAO
     (執行)  │  (監督)
        ╲    │   ╱
         ╲   │  ╱
          CHRO
        (制度建設)

CEO 管不了 CAO 的報告
CAO 管不了 CHRO 的政策撰寫
CHRO 管不了 CEO 的決策
三者互相制衡，只有你能打破僵局
```

## 最終組織架構圖

```
              你（董事長）
             ╱           ╲
       CAO 稽核長      CEO 總經理
       │              ┌──┼──────┬──────┬──────┐
       │             CFO  CIO   COO   CTO   CHRO
       │              │    │     │      │     │
       │           記帳  投資  管家  開發總監  能力管理
       │           預算  監控  行程    │      技能開發
       │           分析  建議  飲食  ┌─┴─┐   組織進化
       │           成本  組合  訂票  工兵群   模型評估
       │           審計  管理       (Sub-Agents)
       │
    獨立監督所有 Agent
    稽核閉環追蹤
    安全/財務/合規稽核（Sub-Agents）
    直接向董事長報告
```

### 治理原則

1. **不設委員會** — 用「顧問團分析」取代「投票表決」
2. **所有重大決策權在你** — Agent 提供分析，你拍板
3. **政策變更必須通知你** — 分三級通知
4. **完整版本追蹤** — changelog + Git 備份
5. **三角制衡** — CEO 執行、CAO 監督、CHRO 制度建設
6. **任何角色都無法審批涉及自己的規範修訂**

### 技術實現要點

- **Full Agent（8 個）**：CEO、CFO、CIO、COO、CTO、CHRO、CAO — 各自獨立 workspace
- **Sub-Agent（臨時）**：開發工兵、稽核官 — 無狀態、做完消失
- **記憶控制**：MEMORY.md ≤200 行 + 每日日誌 + LanceDB 冷記憶
- **治理文件**：AGENTS.md 索引 + policies/ 按需載入
- **成本控制**：模型分級 + CFO 審計 + CAO 煞車
