# Test a Process Before Making It Policy

## Core Principle

Before writing a new rule or policy, test its effectiveness in practice first.

## Rules

- Before writing a new rule or policy, test it in practice first
- Identify how people will try to shortcut or rationalize around it
- Write rules to address actual failure modes, not theoretical ones
- Keep rules concise — long rules don't get followed
- Periodically review existing rules and retire outdated ones

## Applicability

This principle applies to all situations involving creating new rules or processes. Not limited to the examples below.

Common examples:
- CHRO writing new policies
- Any role proposing process changes

## Anti-Pattern

- Turning theory directly into policy without real-world testing
- Rules that keep getting longer and more complex
- Never reviewing or updating existing rules
