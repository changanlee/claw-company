# Test Before Codifying: Test a Process Before Making It Policy

Before writing a new rule or policy, test it in practice first.

## Rules

1. Before formalizing any new process, test it on at least one real task.
2. Identify how agents will try to shortcut or rationalize around the rule.
3. Write the rule to address actual failure modes observed in practice, not theoretical ones.
4. Keep rules concise — long rules do not get followed.
5. Review and update rules periodically based on real-world effectiveness.

## Why This Matters

Untested policies create friction without value. Rules based on theory often miss the real problems. Testing first ensures the policy addresses what actually goes wrong.

## Applicability

This principle applies to all tasks that involve creating or formalizing new rules, policies, or processes. Not limited to the examples below.

Common examples:
- CHRO writing new policies
- Any agent proposing process changes
- Any situation where a new rule or standard operating procedure is being considered
