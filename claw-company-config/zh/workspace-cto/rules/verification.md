# 完成前驗證鐵律

1. **沒有新鮮的驗證證據，不能宣稱完成。** 「我記得之前跑過」不算數。
2. 宣稱成功前，執行以下四步驟：確認驗證命令 → 重新執行 → 讀完整輸出 → 確認輸出支持你的結論。
3. **「應該可以」/「大概沒問題」= 未驗證。** 這些詞出現時，代表你還沒做完。
4. 跑命令、讀輸出、然後才能宣稱結果。順序不可顛倒，步驟不可省略。
5. 適用範圍：測試通過、建置成功、Bug 修復、部署完成、功能交付——全部適用。
6. 驗證必須在**當前環境**執行，不能引用其他環境的結果。
7. 部分通過 ≠ 通過。100 個測試過了 99 個 = 失敗。
8. 驗證證據必須包含在回報中：命令、輸出、結論，三者缺一不可。
9. 如果驗證過程發現新問題，**先修再報**，不能隱瞞或延後。

## 反合理化對照表

| 藉口 | 事實 |
|------|------|
| 「剛剛改的地方很小」 | 小改動 = 小驗證成本，沒有理由跳過 |
| 「測試剛跑過了」 | 「剛跑過」不是「這次跑過」，重新跑 |
| 「只是重構沒改行為」 | 重構正是最容易引入隱性 bug 的時候 |
| 「時間不夠了」 | 沒驗證就交付 = 把風險轉嫁給下游 |
| 「我很確定沒問題」 | 確定不是證據，測試結果才是 |
