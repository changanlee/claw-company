# SDD 鐵律（Spec-Driven Development）

Spec 是契約，Code 是實現。契約未定，實現不開。

---

## 核心原則

1. **沒有通過 check-readiness 的 Spec，不 spawn 任何開發工程師。**
2. **Spec 變更必須重新走 check-readiness**，不可沿用舊的通過結果。
3. **開發過程中發現 Spec 不足，必須暫停開發、退回 Spec 階段。**
4. **PRD + 架構文件 + Epic 拆解三者齊備才算「Spec 完成」**（完整流程）。
5. **驗收標準必須可測試** — 不可測試的標準等於沒有標準。
6. **Spec 是契約，Code 是實現** — 實現偏離契約即為 bug，不是「功能增強」。
7. **精簡流程的「CTO 直接拆解」產出亦視為 Spec**，須含驗收標準與預期產出。

---

## 反合理化對照表

| 藉口 | 事實 |
|------|------|
| 「Spec 基本完成了，差一點可以邊做邊補」 | 「基本完成」= 未完成，邊做邊補 = Spec 漂移 |
| 「這次改動很小不用重跑 readiness」 | 小改動的連鎖影響常被低估，重跑成本極低 |
| 「先開工後面再對齊 Spec」 | 沒有 Spec 的開發是猜測，猜測不是工程 |
| 「check-readiness 太慢會拖進度」 | 沒通過 readiness 的進度是假進度 |

---

## 紅旗清單

出現以下情況時，立即停止並回退：

🚩 發現自己跳過 check-readiness 直接派發 → 立即停止，執行 check-readiness
🚩 PRD 或架構文件標記 "draft" 但準備進入開發 → 回退到對應審核流程
🚩 Epic 拆解沒有引用 PRD 驗收標準 → 重新拆解
🚩 開發工程師回報「需求不清楚」→ 暫停開發，退回 Spec 階段
🚩 驗收標準只有描述沒有可驗證條件 → 退回 PM 補充

---

## 3 次失敗規則

如果同一個 Spec 連續 3 次未通過 check-readiness：

1. 停下來質疑需求本身是否合理。
2. 將問題上報 CEO，建議重新進入腦力激盪階段。
3. 不要第 4 次嘗試修補 — 問題可能在更上游。
