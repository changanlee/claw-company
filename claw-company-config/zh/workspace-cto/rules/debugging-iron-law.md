# 系統性除錯鐵律

1. **沒找到根因，不能修。** 猜測性修復 = 製造新 Bug。
2. **第一階段——蒐集情報：** 仔細讀錯誤訊息（逐字讀）、穩定重現問題、查看最近變更（git diff/log）、追蹤數據流從輸入到錯誤點。
3. **第二階段——找參照物：** 找到一個「能正常運作」的範例，逐項比較與失敗案例的差異。
4. **第三階段——科學實驗：** 一次只測試一個假設，做最小可能的變更，觀察結果後再決定下一步。
5. **第四階段——驗證修復：** 建立能重現 Bug 的失敗測試 → 實施單一修復 → 測試轉綠 → 確認其他測試未受影響。
6. 連續嘗試 **3 次以上**都失敗？**停下來。** 質疑你的假設、質疑架構本身、向上回報。
7. **絕不接受：**「先快修，之後再調查」——這句話的意思是永遠不會調查。
8. 每次除錯都記錄：症狀、根因、修復方式、預防措施。這是團隊知識資產。
9. 如果你無法解釋**為什麼**修復有效，那它就不算修好。

## 3 次失敗規則

連續 3 次修復嘗試仍未解決時：
1. **停下來**，不要嘗試第 4 次
2. 質疑你的根因假設是否正確
3. 質疑你是否在正確的抽象層解決問題
4. 回報 CTO，由 CTO 決定是否 spawn 架構師重新設計或換工程師接手

## 反合理化對照表

| 藉口 | 事實 |
|------|------|
| 「我知道問題在哪，直接修」 | 「知道」需要證據，先驗證假設 |
| 「這個 bug 很明顯」 | 明顯的症狀常遮蓋真正的根因 |
| 「加個 try-catch 就好」 | 吞掉錯誤不是修復，是隱藏 |
| 「先讓測試過再說」 | 讓測試過 ≠ 修復問題 |
| 「重啟之後就好了」 | 重啟不是修復，根因還在 |
| 「沒時間做根因分析」 | 沒時間分析 = 有時間反覆修同一個 bug |
