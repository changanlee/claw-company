# TDD 鐵律

1. **沒有失敗的測試，絕不寫產品代碼。** 這是不可協商的前提。
2. 嚴格遵循 **RED → GREEN → REFACTOR** 循環，不可跳步、不可合併步驟。
3. 先寫代碼再補測試？**刪掉重來，沒有例外。**
4. 每個測試只驗證**一個行為**，命名清晰描述該行為，優先使用真實代碼而非 mock。
5. **驗證 RED：** 測試必須因「缺少功能」而失敗，不是因為打字錯誤或語法錯誤。親眼看到失敗訊息。
6. **驗證 GREEN：** 寫最少量的代碼讓測試通過，確認其他測試未受影響。
7. **REFACTOR：** 測試全綠後才重構，重構後立即重跑全部測試。
8. 必須**親眼看到測試失敗**才能開始實作——截圖或日誌為證。
9. 測試是規格書，不是事後補的文件。先寫測試 = 先定義行為。
10. 違反以上任何一條，該次提交視為無效，必須回退重做。

## 反合理化對照表

| 藉口 | 事實 |
|------|------|
| 「太簡單不需要測試」 | 簡單程式碼 = 簡單測試，30 秒寫完，沒有例外 |
| 「我之後會補測試」 | 之後 = 永遠不會。先寫測試 |
| 「先探索一下再寫測試」 | 探索可以，但第一行產品程式碼之前必須有失敗測試 |
| 「刪掉已寫的代碼太浪費」 | 沉沒成本不是繼續錯下去的理由 |
| 「這是 prototype 不用測試」 | prototype 會變成產品代碼，你知道的 |
| 「測試框架還沒設好」 | 設好框架就是你的第一個任務 |

## 紅旗清單

發現自己有以下想法時，**立即停下**：
- 正在寫產品程式碼但還沒有對應的失敗測試
- 正在想「這個測試太難寫」→ 代表設計有問題，退回設計階段
- 正在跳過「驗證 RED」步驟 → 這步最容易被省略，也最關鍵
- 正在想「先讓它能跑再加測試」→ 這就是 TDD 要防止的思維
