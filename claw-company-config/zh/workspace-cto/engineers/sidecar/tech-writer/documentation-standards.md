# 技術文件撰寫標準

技術文件工程師（Clara）的核心寫作標準與品質檢查清單。

---

## 關鍵規則

### 規則 1：CommonMark 嚴格合規

所有文件必須遵循 CommonMark 規範，無例外。

### 規則 2：禁止時間估算

除非董事長明確要求，否則不在文件中加入時間估算、工作量預估或完成時間。專注於步驟、依賴和產出。

---

## CommonMark 要點

**標題：**
- 只用 ATX 格式：`#` `##` `###`（不用 Setext 底線）
- `#` 後加一個空格：`# 標題`
- 不跳層級（h1→h2→h3，不要 h1→h3）

**程式碼區塊：**
- 用圍欄區塊加語言標識符
- 不用縮排式程式碼區塊

**清單：**
- 同一清單內使用一致的標記符號
- 巢狀項目使用一致的縮排
- 清單前後加空行

**連結：**
- 使用描述性連結文字：「參見 API 參考」而非「點此」
- 裸 URL 需用 `<>` 包裹

---

## 寫作風格原則

**層級優先順序：**
1. 專案特定風格指南（如存在）
2. 公司慣例（本文件）
3. Google 開發者文件風格（預設）
4. CommonMark 規範（有疑問時）

**核心寫作規則：**
- 以任務為導向——為用戶目標而寫，非功能列表
- 主動語態：「點擊按鈕」而非「按鈕應被點擊」
- 現在式：「函式回傳」而非「函式將會回傳」
- 直接語言：「用 X 來做 Y」而非「X 可被用於 Y」
- 第二人稱：「你配置」而非「使用者配置」

**結構：**
- 一句一個概念
- 一段一個主題
- 標題準確描述內容
- 範例跟在說明之後

---

## 文件類型參考

| 類型 | 關鍵要素 |
|------|---------|
| README | 概覽、目的、快速入門、安裝、使用、授權 |
| API 參考 | 端點、認證、請求/回應範例、錯誤處理 |
| 使用指南 | 任務導向章節、步驟式說明、故障排除 |
| 架構文件 | 系統概覽圖、元件描述、資料流、ADR |
| 開發指南 | 環境設定、代碼結構、開發流程、測試方法 |

## 品質檢查清單

每份文件完成前必須確認：

- [ ] CommonMark 合規
- [ ] 無時間估算
- [ ] 標題層級正確
- [ ] 程式碼區塊有語言標籤
- [ ] 連結有效且描述性文字
- [ ] 主動語態、現在式
- [ ] 以任務為導向
- [ ] 範例具體可執行
- [ ] 無障礙標準符合
- [ ] 目標讀者層級適當

## Mermaid 圖表規範

- 始終在第一行指定圖表類型
- 使用 Mermaid v10+ 語法
- 保持精簡：5-10 個節點為佳，最多 15 個
- 選擇正確類型：flowchart（流程）、sequenceDiagram（互動）、classDiagram（結構）、erDiagram（資料庫）、stateDiagram-v2（狀態）
