# PRD 撰寫標準

Reed（產品經理）的領域知識參考，spawn 時自動載入。

---

## Jobs-to-be-Done 框架

每個需求都應該能回答：

> 當 {情境} 時，{用戶角色} 想要 {動作}，以便 {預期結果}。

### JTBD 層級

| 層級 | 說明 | 範例 |
|------|------|------|
| 功能性 | 用戶要完成的具體任務 | 「快速登入系統」 |
| 情感性 | 用戶想要的感受 | 「不想記住複雜密碼」 |
| 社會性 | 用戶想被如何看待 | 「看起來技術熟練」 |

**PRD 中的應用**：每個功能需求至少對應一個功能性 JTBD，高優先級功能應同時涵蓋情感性層面。

## 用戶故事格式

### 標準格式

```
作為 {用戶角色}，
我希望 {功能描述}，
以便 {商業價值/用戶價值}。
```

### 驗收標準格式（Given-When-Then）

```
場景：{場景名稱}
  假設（Given）{前置條件}
  當（When）{觸發動作}
  那麼（Then）{預期結果}
```

### 驗收標準可測試性檢核表

每條驗收標準必須通過以下檢查：

- [ ] **明確性**：是否有唯一解讀？（不含「適當的」「合理的」等模糊詞）
- [ ] **可觀察性**：結果是否可被外部觀察或測量？
- [ ] **邊界條件**：是否定義了邊界情況（空值、極大值、並發）？
- [ ] **錯誤路徑**：是否定義了錯誤情況的預期行為？
- [ ] **可自動化**：是否能轉化為自動化測試？

**紅旗詞彙**（出現即需修改）：
- 「應該」→ 改為「必須」或刪除
- 「快速」→ 改為具體數值（如 <200ms）
- 「友好的」→ 改為具體 UI 行為描述
- 「等等」→ 列出完整清單

## PRD 結構規範

### 必備段落

| 段落 | 必填 | 說明 |
|------|------|------|
| 執行摘要 | ✅ | 一段話說明產品目標與價值 |
| 用戶角色 | ✅ | 主要用戶 persona（含痛點與目標） |
| 功能需求 | ✅ | 按優先級排序，每條含驗收標準 |
| 非功能需求 | ✅ | 效能、安全、可用性等 |
| 成功指標 | ✅ | 可量化的 KPI |
| 範圍定義 | ✅ | 明確列出「不做什麼」（Out of Scope） |
| 用戶旅程 | 選填 | 主要流程的步驟描述 |
| 競品分析 | 選填 | 市場現有解決方案對比 |

### 優先級框架（MoSCoW）

| 等級 | 定義 | PRD 中的標記 |
|------|------|-------------|
| Must Have | 沒有就無法上線 | 🔴 P0 |
| Should Have | 重要但可延後 | 🟡 P1 |
| Could Have | 錦上添花 | 🟢 P2 |
| Won't Have | 明確不做 | ⚪ Out of Scope |

## PRD 品質自檢

完成 PRD 後，PM 應逐項確認：

1. 每個功能需求都有至少一條 Given-When-Then 驗收標準
2. 所有驗收標準通過「可測試性檢核表」
3. 非功能需求有具體數值（不是「快」而是「<200ms」）
4. Out of Scope 段落不為空
5. 成功指標可量化且有基線對比
6. 用戶角色來自研究或訪談，不是臆測
