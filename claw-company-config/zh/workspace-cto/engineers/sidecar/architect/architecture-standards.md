## ADR（架構決策記錄）格式

每個架構決策必須記錄為以下結構：

```markdown
### ADR-{NNN}: {決策標題}

**狀態：** 提議 | 接受 | 取代 | 廢棄
**日期：** YYYY-MM-DD
**決策者：** {角色}

#### 問題
{需要做出決策的背景與問題描述}

#### 選項
1. {選項 A} — {優點} / {缺點}
2. {選項 B} — {優點} / {缺點}
3. {選項 C} — {優點} / {缺點}

#### 決策
選擇 {選項 X}，因為 {理由}。

#### 後果
- 正面：{預期收益}
- 負面：{已知代價}
- 風險：{需要監控的風險}
```

## Boring Technology 原則

引用自 Dan McKinley「Choose Boring Technology」：

1. **每個專案只有 3 個創新額度** — 把創新用在真正需要的地方，其餘用成熟技術
2. **成熟技術的隱性優勢** — 已知的 bug、豐富的文件、龐大的社群支援、招聘容易
3. **新技術的隱性成本** — 未知的 bug、學習曲線、生態不完善、可能被放棄
4. **選擇標準** — 除非新技術能解決成熟技術無法解決的核心問題，否則選成熟方案

## 可擴展設計模式

### 分層架構（推薦起步模式）

```
Presentation → Application → Domain → Infrastructure
```

- 依賴方向：只能向下依賴，不可反轉
- Domain 層不依賴任何框架

### 擴展決策點

| 指標 | 閾值 | 擴展策略 |
|------|------|---------|
| 用戶數 | >10K DAU | 考慮讀寫分離 |
| API QPS | >1000 | 考慮快取層 |
| 資料量 | >10GB | 考慮分片或分區 |
| 團隊規模 | >3 服務 | 考慮 API Gateway |

**原則**：先跑起來，再根據瓶頸數據決定擴展方向。過早優化是萬惡之源。

## 技術選型決策框架

評估維度（1-5 分）：

| 維度 | 權重 | 說明 |
|------|------|------|
| 成熟度 | 30% | 版本穩定性、生產案例數 |
| 社群生態 | 20% | 文件品質、套件豐富度、活躍度 |
| 團隊熟悉度 | 20% | 現有團隊經驗、學習曲線 |
| 適配度 | 20% | 與需求的匹配程度 |
| 長期維護 | 10% | 背後組織、商業模式、活躍開發者 |

**紅旗**：任一維度低於 2 分 → 需要在 ADR 中特別說明風險緩解方案。
