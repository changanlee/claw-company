# 測試標準與策略

QA 工程師（Vera）的核心測試標準，融合 TEA（Test Engineering & Architecture）模組精華。

---

## 測試架構原則

### 原則 1：架構先行

先設計測試架構與自動化框架，再寫個別測試案例。測試架構決定了測試的可維護性和擴展性。

### 原則 2：風險驅動

測試優先順序基於風險評估：

1. **關鍵路徑** — 核心業務流程，失敗即系統不可用
2. **邊界條件** — 極值、空值、溢出、型別邊界
3. **異常處理** — 錯誤恢復、逾時、網路中斷
4. **一般場景** — 正常使用情境

### 原則 3：品質閘門不可妥協

每個階段都有明確的品質門檻：
- 測試覆蓋率達到約定標準
- 關鍵路徑 100% 通過
- 零已知 Critical/High 缺陷
- 效能基準在可接受範圍內

---

## 測試金字塔

```
        /  E2E  \        ← 少量：驗證端到端流程
       /  整合測試  \      ← 適量：驗證元件互動
      / 單元測試      \    ← 大量：驗證個別函式邏輯
```

**比例建議：** 70% 單元 / 20% 整合 / 10% E2E

---

## 測試設計模式

### AAA 模式（Arrange-Act-Assert）

每個測試案例遵循：
1. **Arrange** — 準備測試資料與環境
2. **Act** — 執行被測功能
3. **Assert** — 驗證結果符合預期

### 測試命名

格式：`test_[被測功能]_[情境]_[預期結果]`

範例：`test_login_invalidPassword_returnsError`

### 測試獨立性

- 每個測試案例獨立運行，不依賴其他測試的執行順序
- 測試之間不共享可變狀態
- 使用 setup/teardown 管理測試環境

## 測試類型參考

| 類型 | 目的 | 工具參考 |
|------|------|---------|
| 單元測試 | 驗證個別函式/方法邏輯 | Jest, pytest, JUnit |
| 整合測試 | 驗證元件間互動 | Supertest, TestContainers |
| E2E 測試 | 驗證端到端流程 | Playwright, Cypress |
| 效能測試 | 驗證回應時間與吞吐量 | k6, Artillery |
| 安全測試 | 識別安全漏洞 | OWASP ZAP, Snyk |
| 無障礙測試 | 驗證可訪問性 | axe, Lighthouse |

## 缺陷分級

| 等級 | 定義 | 處理 |
|------|------|------|
| Critical | 系統崩潰、資料遺失、安全漏洞 | 立即修復，阻擋發布 |
| High | 核心功能失效、無替代方案 | 當前 Sprint 修復 |
| Medium | 功能異常但有替代方案 | 排入後續 Sprint |
| Low | 外觀問題、文字錯誤 | Backlog |

---

## CI/CD 品質閘門

建議的品質閘門檢查點：

1. **PR 合併前** — 單元測試全通過、Lint 無錯誤
2. **Staging 部署前** — 整合測試通過、安全掃描無 Critical
3. **Production 部署前** — E2E 測試通過、效能基準達標
4. **部署後** — 冒煙測試、監控告警正常

---

## 非功能性需求（NFR）評估

評估項目清單：

- **效能**：回應時間 p95、吞吐量、並發用戶數
- **可靠性**：可用率目標、故障恢復時間（RTO/RPO）
- **安全性**：OWASP Top 10、認證機制、資料加密
- **可擴展性**：水平/垂直擴展策略
- **可維護性**：代碼複雜度、測試覆蓋率、文件完整性
