# 步驟 3：架構決策記錄

**進度：步驟 3 / 共 5 步** — 下一步：基礎設施策略

## 目標

以 ADR（Architecture Decision Record）格式記錄所有關鍵架構決策，確保決策可追溯。

## 執行規則

- 📖 讀完整個步驟檔案再開始行動
- 🚫 不預讀後續步驟
- 🚫 不跳步、不合併步驟

## 執行指令（按順序，不可跳步）

### 1. 識別關鍵決策

從步驟 2 的設計中，識別所有需要記錄的架構決策：

- 架構風格選擇（monolith / microservices / serverless）
- 技術棧選擇（程式語言、框架、資料庫）
- 通訊協議選擇
- 認證/授權方案選擇
- 部署策略選擇
- 其他有多個可行方案的決策

### 2. 撰寫 ADR

每個決策使用以下格式：

```markdown
### ADR-{序號}：{決策標題}

**狀態**：Accepted

**問題**：
{需要做決策的問題描述}

**選項**：
1. {選項 A} — {優點} / {缺點}
2. {選項 B} — {優點} / {缺點}
3. {選項 C} — {優點} / {缺點}（如有）

**決策**：
選擇 {選項 X}

**理由**：
{為什麼選擇這個選項，考量了哪些因素}

**後果**：
- 正面：{帶來的好處}
- 負面：{帶來的代價或限制}
- 風險：{潛在風險}
```

### 3. 更新 frontmatter

將 `step-03-adr` 加入 `steps-completed`。

## 完成條件

- ✅ 所有關鍵架構決策已識別
- ✅ 每個決策都有完整的 ADR（問題/選項/決策/理由/後果）
- ✅ 決策之間無矛盾
