# 步驟 3：產出就緒性報告

**進度：步驟 3 / 共 3 步** — 最後一步

## 目標

根據驗證結果產出就緒性報告，給出明確的就緒性判定。

## 執行規則

- 📖 讀完整個步驟檔案再開始行動
- 🚫 不預讀後續步驟
- 🚫 不跳步、不合併步驟

## 執行指令（按順序，不可跳步）

### 1. 計算就緒性分數

統計驗證結果：

- ✅ 通過的項目數
- ⚠️ 部分通過的項目數
- ❌ 未通過的項目數

### 2. 判定就緒性等級

根據驗證結果給出判定：

| 等級 | 條件 | 說明 |
|------|------|------|
| ✅ Ready | 0 個 ❌，⚠️ ≤ 2 個 | 可以開始開發 |
| ⚠️ Needs Work | 0 個 ❌，⚠️ > 2 個 | 需要補充後才能開始 |
| ❌ Not Ready | 任何 ❌ | 有重大缺口，需返工 |

### 3. 撰寫就緒性報告

```markdown
# 開發就緒性報告

**日期**：{today}
**判定**：{✅ Ready / ⚠️ Needs Work / ❌ Not Ready}
**分數**：{通過數} / {總項目數}

## 文件完整性

| 文件 | 狀態 | 通過率 | 主要問題 |
|------|------|--------|---------|
| PRD | ✅/⚠️/❌ | X/Y | {問題描述} |
| UX 設計 | ✅/⚠️/❌ | X/Y | {問題描述} |
| 架構文件 | ✅/⚠️/❌ | X/Y | {問題描述} |
| Epic/Story | ✅/⚠️/❌ | X/Y | {問題描述} |

## 跨文件一致性

{一致性驗證結果}

## 需要修正的項目

{按優先級列出需要修正的項目}

## 建議

{下一步建議}
```

### 4. 寫入產出目錄

將就緒性報告寫入 `{{INSTALL_DIR}}/workspace-cto/output/planning/readiness-report.md`。

### 5. 回報 CTO

透過 `announce` 將就緒性報告回報 CTO：

- 明確告知判定結果（Ready / Needs Work / Not Ready）
- 如為 Needs Work 或 Not Ready，列出需要修正的項目
- 建議的行動方案

## 完成條件

- ✅ 就緒性分數已計算
- ✅ 就緒性等級已判定
- ✅ 報告已撰寫完成
- ✅ 檔案已寫入產出目錄
- ✅ 已回報 CTO

## 下一步

此為最後一步，就緒性檢查流程完成。等待 CTO 回覆。
