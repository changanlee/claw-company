# 步驟 2：規模評估

**進度：步驟 2 / 共 5 步**

## 目標

根據需求的複雜度與風險，判斷應走精簡流程還是完整流程，決定後續任務拆解與審查的深度。

## 執行規則

- 📖 讀完整個步驟檔案再開始行動
- 🚫 不預讀後續步驟
- 🚫 不跳步

## 執行指令

### 1. 評估需求規模

根據技術設計文件，評估需求的複雜度並向上級（CEO 或董事長，取決於 session 來源）建議流程：

> 「這個需求建議走**精簡/完整**流程，預估需要 N 個工程師 spawn。」

### 2. 精簡流程

適用於：小功能、Bug 修復、直覺明確的任務。

- 跳過 PRD 和架構設計階段。
- CTO 直接拆解任務並派發。
- 後續步驟 3 走精簡路徑，步驟 5 走精簡審查。
- **Spawn 上限：4 個**（Dev + CR + 退回重做 + buffer）

### 3. 完整流程

適用於：複雜產品、多元件系統、新架構、高風險變更。

- 完整執行 PRD → 架構設計 → 任務拆解 → 開發 → 審查。
- 後續步驟 3 走完整路徑，步驟 5 走完整審查。
- **Spawn 上限：12 個**（PM + Architect + SM + readiness + Dev + SR + CR + TEA + 退回重做）

### 4. 記錄決策

將流程選擇（精簡/完整）+ 預估 spawn 數記錄到 `status.md`，後續步驟依此分支執行。CAO heartbeat 會掃描此記錄進行 spawn 監控。

## 完成標準

- [ ] 已向上級說明建議的流程規模、理由及預估 spawn 數
- [ ] 上級已確認流程選擇（精簡或完整）— 黃燈，CEO 可審批
- [ ] 已將流程決策與預估 spawn 數記錄到 status.md
