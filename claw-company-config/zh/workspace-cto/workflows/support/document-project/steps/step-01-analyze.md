# 步驟 1：分析文件現狀

**進度：步驟 1 / 共 3 步** — 下一步：撰寫文件

## 目標

分析專案代碼庫和現有文件，識別文件缺口，建立文件撰寫計畫。

## 執行規則

- 📖 讀完整個步驟檔案再開始行動
- 🚫 不預讀後續步驟
- 🚫 不跳步、不合併步驟

## 執行指令

### 1. 掃描專案結構

瀏覽專案目錄結構，了解：

- 專案整體架構
- 主要模組和元件
- 技術棧（語言、框架、工具）
- 現有的文件檔案

### 2. 評估現有文件

檢查已有的文件：

- README.md 是否存在且完整
- API 文件是否存在
- 開發指南是否存在
- 代碼內註解品質

### 3. 識別文件缺口

列出缺少或不完整的文件：

| 文件類型 | 狀態 | 優先級 |
|---------|------|--------|
| README | 缺少/不完整/完整 | 高/中/低 |
| API 文件 | 缺少/不完整/完整 | 高/中/低 |
| 開發指南 | 缺少/不完整/完整 | 高/中/低 |
| 架構說明 | 缺少/不完整/完整 | 高/中/低 |
| 部署指南 | 缺少/不完整/完整 | 高/中/低 |

### 4. 建立撰寫計畫

按優先級排列文件撰寫順序：

1. README（專案入口）
2. API 文件（開發者需要）
3. 開發指南（新人上手）
4. 其他文件

## 完成條件

- ✅ 專案結構已了解
- ✅ 現有文件已評估
- ✅ 文件缺口已識別
- ✅ 撰寫計畫已建立
