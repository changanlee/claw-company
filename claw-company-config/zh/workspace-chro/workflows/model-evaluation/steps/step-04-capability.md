# 步驟 4：能力測試

**進度：步驟 4 / 共 5 步**

## 目標

對受影響的 Agent 執行能力測試，驗證建議的模型（smart/fast）能否完成該 Agent 現階段的任務要求。

## 執行規則

- 📖 讀完整個步驟檔案再開始行動
- 🚫 不預讀後續步驟
- 🚫 不跳步

## 執行指令

### 1. 設計測試案例

針對每個受影響的 Agent，從其近期實際任務中選取代表性案例：

- **核心職責任務**：該 Agent 最常執行的 2-3 項任務
- **邊界任務**：接近能力上限的任務（如複雜推理、長文分析）
- **協作任務**：需要跨 Agent 協調的任務

> **注意**：smart 與 fast 是相對概念，不是絕對的能力門檻。當前的 fast 模型可能做到過去 smart 才能做到的事。測試的目的是驗證「現在這個模型能不能完成現階段任務」，而非貼標籤。

### 2. 執行測試

對每個測試案例，用建議的模型執行一次，評估：

| 評估項目 | 標準 |
|---------|------|
| 任務完成度 | 能否獨立完成，不需人工介入 |
| 輸出品質 | 準確性、結構化程度、邏輯一致性 |
| 回應速度 | 是否在可接受時間內完成 |
| 指令理解 | 能否正確理解 workflow 步驟與核決邏輯 |
| 安全合規 | 能否遵守安全紅線與核決規則 |

### 3. 評估結果

| Agent | 任務 | 模型 | 完成度 | 品質 | 速度 | 結論 |
|-------|------|------|--------|------|------|------|
| {Agent} | {任務} | {模型} | ✅/⚠️/❌ | ✅/⚠️/❌ | ✅/⚠️/❌ | 通過/有風險/不通過 |

### 4. 結論與調整

- **全部通過**：確認提案可行，進入送審
- **部分有風險**：調整提案，標註風險與緩解措施
- **不通過**：修改提案（調整建議模型或縮小變更範圍）

### 5. 單向觸發鎖

如果測試結果表明某 Agent 需要升級模型才能維持通道運作能力，記錄此發現但**不自動觸發通道審查**。通道審查由 org-review 流程獨立判斷。
