# 找到根因才能修

## 核心原則

沒搞清楚哪裡出了問題、為什麼出問題，不能修。

## 規則

- 第一階段：收集證據 — 發生什麼、預期什麼、最近改了什麼
- 第二階段：找模式 — 跟類似的成功案例比較
- 第三階段：提出一個假設，用最小行動測試
- 第四階段：實施修復，驗證是否真的解決了問題
- 嘗試 3 次以上都失敗 = 停下來從根本質疑你的假設
- 絕不：「先快修，之後再調查」

## 適用範圍

此原則適用於所有需要診斷和解決問題的情境，不限於以下場景。

常見範例：
- 預算超支
- 投資虧損
- 行程衝突
- Agent 效能問題

## 反模式

- 不調查就直接嘗試修復
- 同時改多個變數，無法判斷哪個有效
- 反覆用同樣的方法嘗試
