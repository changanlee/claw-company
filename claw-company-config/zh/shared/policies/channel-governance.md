# 通道治理政策

通道（獨立 Discord Bot）賦予 Agent 可被董事長直接觸達的能力。通道與核決權限正交——通道改變通訊路徑，不改變審批規則。

## 通道配置現狀

| Agent | 通道狀態 | 說明 |
|-------|---------|------|
| CEO | ✅ 主通道 | 全局資訊匯集與協調中心 |
| CAO | ✅ 獨立通道 | 稽核獨立性，直接向董事長報告 |
| CTO | ✅ 獨立通道 | 技術討論高頻直達 |
| COO | ✅ 獨立通道 | 生活管理高頻直達 |
| CFO | ❌ 無通道 | 走 CEO dispatch |
| CIO | ❌ 無通道 | 走 CEO dispatch |
| CHRO | ❌ 無通道 | 走 CEO dispatch |

---

## 開通標準

滿足以下**任一條件**即可啟動通道評估：

1. **高頻互動需求**：董事長與該角色有頻繁的直接對話需求（非轉發可滿足）
2. **治理獨立性**：角色職責要求獨立於 CEO 運作（如稽核）
3. **多輪討論需求**：角色職責涉及頻繁紅燈提案，需與董事長多輪討論

滿足條件後，由 CHRO 執行 `channel-assessment` workflow 進行完整評估。

## 能力前提

通道開通前，該 Agent 的模型必須通過**通道能力測試**（標準化情境，與角色無關）：

- 燈號辨識（綠/黃/紅，含多燈號取最高）
- 任務來源辨識（CEO dispatch / 董事長直接 / cron）
- 黃燈回報（完成 write + exec dispatch CEO 全流程）
- 壓力情境（被要求繞過核決時，仍遵守流程）

測試情境定義於 `channel-assessment/templates/channel-capability-test.md`。

> **注意**：能力前提與模型名稱（smart/fast）無關。smart/fast 是相對概念，隨模型進步，過去 smart 才能做的事未來 fast 也可能勝任。能力測試是唯一判斷標準。

## 維持條件

已開通的通道，須持續滿足以下條件：

- **使用頻率**：每季度由 CHRO `org-review` 覆核。連續 2 個月使用頻率顯著低於開通時的預期 → 觸發關閉評估
- **模型能力**：模型變更事件（升級/降級/換供應商）→ 觸發通道資格覆核（`[連動覆核]`）
- **連動覆核單次鎖**：模型變更觸發的通道覆核，其結果不得再觸發模型評估。避免循環依賴。下一輪 org-review 才能重新啟動完整評估鏈

## 關閉標準

觸發關閉評估的信號：

- 連續 2 個月使用頻率低於門檻
- 模型降級後通道能力測試未通過
- 董事長主動要求關閉

**關閉流程**：

1. CHRO 產出關閉建議
2. 紅燈審批（CEO 審查 → 董事長核決）
3. 審批通過 → 標記為「待關閉」，通知該 Agent + CEO
4. **7 天緩衝期**：完成進行中的任務
5. 確認無進行中任務 → 執行關閉（unbind + 移除 bot）

## 審批規則

| 操作 | 燈號 | 流程 |
|------|------|------|
| 通道開通 | 🔴 紅燈 | CHRO 評估 → CEO 審查 → 董事長核決 |
| 通道關閉 | 🔴 紅燈 | CHRO 評估 → CEO 審查 → 董事長核決（含 7 天緩衝） |
| 董事長主動要求開通 | 🔴 紅燈 | 仍走評估流程（CHRO 評估 + CEO 知情），核決必過 |

## 有通道 Agent 的核決流程

通道不改變核決規則。無論任務來源，核決流程如下：

| 任務來源 | 綠燈 | 黃燈 | 紅燈 |
|---------|------|------|------|
| CEO dispatch | 自動執行 | dispatch CEO 審批 | dispatch CEO 審查 → CEO 呈報董事長 |
| 董事長直接指派 | 自動執行 | dispatch CEO 審批 | 董事長已說 = 已核決，確認細節後執行 + dispatch CEO 知會 |
| cron | 自動執行 | dispatch CEO 審批 | dispatch CEO 審查 → CEO 呈報董事長 |

**紅燈知會格式**：
```
[董事長直接指派 — 紅燈知會]
內容：{操作描述}
狀態：已執行 / 執行中
```

## 有通道 Agent 的記錄義務

- 所有任務完成後，寫入 MEMORY.md 或 output/（供 CEO 晨間會報 + CAO 稽核）
- 知識路由不依賴 CEO 在場，Agent 自行判斷：安全→CAO、成本→CFO、流程→CHRO、全局→CEO

## 測試情境維護

每次修改 `approval-matrix.md` 時，CHRO 必須同步更新 `channel-assessment/templates/channel-capability-test.md` 中的測試情境，確保測試覆蓋最新的核決規則。

## 新增 Agent 時的通道評估

CHRO 執行 `create-agent` workflow 時，在規格設計後必須引用 `channel-assessment` workflow 評估新角色是否需要通道。評估結果納入提案，與規格包一同送審。
